  <action>
    Add capture analysis to `QwikTransform` in transform.rs. This is the most complex part of the optimizer.

    **The 8 capture categories (from spec CONV-03):**

    1. **Module-level imports** -- Variables imported via `import { X }` from any module. These are NOT captures. They become re-emitted `import` statements in the segment module.

    2. **Module-level declarations** -- `const`, `function`, `class` declared at file root scope. These are NOT captures. They become **self-imports**: `import { X } from "./module_stem"` in the segment module. This is the CRITICAL reclassification behavior (Pitfall 2).

    3. **Outer locals** -- Variables declared in an enclosing function scope (not file root). These ARE captures. They go into the `captures` array in the QRL wrapper: `.w([outerVar])`.

    4. **Inner locals** -- Variables declared INSIDE the `$()` callback body. These are NOT captures (they're local to the segment).

    5. **Loop variables** -- Variables from `for`, `for-of`, `for-in` loop declarations. These ARE captures (they close over the loop iteration value).

    6. **Shadowed bindings** -- A variable name exists in both outer and inner scope. The inner binding shadows the outer. NOT a capture (uses the local binding).

    7. **Destructured params** -- Parameters destructured in the `$()` callback. NOT captures (they're parameters of the segment function).

    8. **Hoisted functions** -- Function declarations inside the `$()` body that are hoisted. NOT captures (they're local).

    **Implementation approach using OXC Scoping (per D-09):**

    When a `$()` call is detected (in `enter_call_expression`), record the scope ID of the callback function body. During traversal of the callback body, for each `IdentifierReference`:

    1. Use `ctx.scoping().symbol_table()` to find the symbol's declaration scope
    2. Compare the declaration scope with the callback's scope:
       - If declared INSIDE callback scope (or nested child scope): Category 4/7/8 (inner local/destructured param/hoisted fn) -- NOT a capture
       - If declared at MODULE scope (scope depth 0): Check GlobalCollect:
         - If in `collect.imports`: Category 1 (module import) -- record as needed_import for segment
         - If in `collect.root`: Category 2 (module declaration) -- record as self-import
       - If declared in BETWEEN scope (not module, not inside callback): Category 3/5 (outer local/loop var) -- IS a capture
    3. For unresolved references (no symbol in scope tree), check GlobalCollect.imports as fallback

    **Self-import reclassification (CRITICAL -- Pitfall 2):**
    After collecting all references, post-process:
    - For each capture candidate, check if it exists in `GlobalCollect.root`
    - If yes: reclassify from "capture" to "self-import" with source `"./{module_stem}"`
    - This is the single most impactful behavioral distinction -- 46 runtime-breaking deviations in Jack's early impl

    **SegmentRecord updates:**
    Add fields to `SegmentRecord`:
    ```rust
    pub captures: Vec<String>,            // actual capture variable names
    pub needed_imports: Vec<ImportInfo>,   // imports the segment needs (from categories 1 & 2)
    pub self_imports: Vec<String>,         // module-level decls referenced by segment
    pub has_captures: bool,               // true if captures vec is non-empty
    ```

    **Edge case handling (CAPTURE-EDGE-01 through 16):**
    - CAPTURE-EDGE-01: TypeScript type-only imports are never captures (skip `ImportKind::Type`)
    - CAPTURE-EDGE-02: `typeof` expressions -- the reference is still a capture if the variable is outer-local
    - CAPTURE-EDGE-03: Nested `$()` calls -- inner $ captures from outer $ body are captures of the OUTER segment
    - Handle all 16 edge cases as documented in the spec

    **CRITICAL anti-patterns to avoid (from research Pitfalls):**
    - Do NOT use wildcard `_ => {}` on BindingPattern matches -- handle all 4 variants (Pitfall 3)
    - Do NOT bail on parse errors in capture analysis -- only bail if `panicked` (Pitfall 4)
    - Do NOT duplicate SemanticBuilder's scope tracking with manual scope stacks (anti-pattern from research)

    Add QRL wrapping to `exit_expression` in QwikTransform. When a `$()` call finishes traversal (exit), replace it with the appropriate QRL wrapper.

    **QRL wrapping rules (CONV-02):**

    For each detected `$()` call (from dollar detection in Plan 04):

    1. **Determine wrapper function name:** Use `dollar_to_qrl_name()` from words.rs:
       - `$()` -> `qrl()`
       - `component$()` -> `componentQrl()`
       - `useTask$()` -> `useTaskQrl()`
       - etc.

    2. **Build segment reference:** Generate the import path and export name for the extracted segment:
       - `qrl(() => import("./path_to_segment"), "segmentExportName")`
       - The segment path and name come from the SegmentRecord created during dollar detection

    3. **Build QRL call expression:**
       For the **Segment** entry strategy (default):
       ```javascript
       qrl(() => import("./seg_canonical_filename"), "seg_name")
       ```
       If captures exist, add `.w([capture1, capture2])`:
       ```javascript
       qrl(() => import("./seg_canonical_filename"), "seg_name").w([capturedVar1, capturedVar2])
       ```

       For the **Inline** entry strategy:
       ```javascript
       inlinedQrl(callback_body, "seg_name", [capturedVar1])
       ```

    4. **PURE annotation (CONV-08):**
       - Add `/*#__PURE__*/` comment ONLY before `componentQrl()` calls
       - Do NOT add to `useTaskQrl`, `serverQrl`, or any other wrapper -- they have side effects (Pitfall 6)
       - The PURE annotation uses OXC's leading comment API on the call expression

    5. **Noop QRL (CONV-14):**
       - If the `$()` callback body is empty (no statements) or only contains a return undefined:
       - Replace with `_noopQrl()` (or `_noopQrlDEV()` in dev mode)
       - Set import flag: `self.needs_noop_qrl_import = true`

    6. **Sync$ (CONV-13):**
       - If the detected call is `sync$()`:
       - Replace with `_qrlSync(stringified_body)` where the body is serialized as a string
       - Do NOT extract a segment for sync$ calls

    7. **Dev mode variants:**
       - In Dev/Hmr emit mode: use `qrlDEV`/`inlinedQrlDEV` instead of `qrl`/`inlinedQrl`
       - Dev variants include extra location info (file, line, column)

    **AST construction using OXC AstBuilder:**
    - Use `ctx.ast` (the AstBuilder available from TraverseCtx) to allocate new AST nodes in the arena
    - Build CallExpression nodes with the correct arguments
    - For member expressions (`.w()`), build a MemberExpression with computed=false

    **Integration with capture analysis:**
    - Pop the SegmentScope from `segment_stack` in `exit_expression`
    - Use the collected captures from capture analysis (Task 1) to build the `.w()` call
    - Mark the SegmentRecord with `has_captures` and the capture list

    - Set `needs_qrl_import = true` when a `qrl()` call is generated
    - Set `needs_inlined_qrl_import = true` when an `inlinedQrl()` call is generated
    - Track all needed synthetic imports for batch application in `exit_program` (Plan 07)
