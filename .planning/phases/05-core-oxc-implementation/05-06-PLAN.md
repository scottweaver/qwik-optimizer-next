    Implement CONV-04: Props destructuring transformation.

    **What it does:** Transforms destructured component$ callback parameters from destructuring syntax to `_rawProps` member access patterns. This enables signal forwarding -- the runtime needs explicit property access calls rather than destructured locals.

    **Rules (from spec):**
    1. **Trigger:** Only applies inside `component$()` callbacks where the first parameter uses destructuring (`{prop1, prop2}`)
    2. **Basic transform:** `({name, age}) => ...` becomes `(_rawProps) => { const name = _rawProps.name; const age = _rawProps.age; ... }`
    3. **String-keyed props (colon syntax):** `{"my-prop": myProp}` becomes `const myProp = _rawProps["my-prop"]` (computed member access)
    4. **Default values:** `({name = "default"}) => ...` becomes `const name = _rawProps.name ?? "default"` (nullish coalescing, NOT logical OR)
    5. **Rest props:** `({name, ...rest})` becomes `const name = _rawProps.name; const rest = _restProps(_rawProps, ["name"])` -- requires `_restProps` import
    6. **Nested destructuring:** `({config: {x, y}})` becomes `const {x, y} = _rawProps.config` (only top-level is transformed)
    7. **Ordering:** Props destructuring MUST run as a pre-pass BEFORE capture analysis. The `_rawProps` identifier and generated const bindings must be visible to capture analysis.

    **Implementation:**
    - Can be implemented as a method on QwikTransform called in `enter_call_expression` when a `component$` is detected
    - OR as a separate pre-pass using VisitMut
    - The approach that integrates with the Traverse is preferred (check during component$ detection, transform the function params before entering the body for capture analysis)
    - Use OXC AstBuilder to create replacement VariableDeclaration nodes
    - For `_restProps`, add to import tracker: `self.needs_rest_props_import = true`

    Update `transform.rs` to integrate props destructuring into the component$ detection path.
    Update `lib.rs` to add the module.
  </action>

    **jsx_transform.rs -- CONV-06: JSX Transform**

    JSX elements are converted to `_jsxSorted()` or `_jsxSplit()` function calls. The choice depends on whether the element has any dynamic props (expressions that could change).

    Key rules:
    1. **Static-only elements** -> `_jsxSorted(tag, staticProps, key, flags, devInfo)`
    2. **Elements with dynamic props** -> `_jsxSplit(tag, {staticProps}, [{dynamicPropName, dynamicPropValue}], key, flags, devInfo)`
    3. **Prop classification:**
       - Static: string literals, number literals, boolean literals, null
       - Dynamic: everything else (expressions, function calls, identifiers that aren't constants)
    4. **Special prop handling:**
       - `className` and `class` both normalize to `class` in the output
       - `bind:value` -> expands to value prop + onInput$ handler
       - `bind:checked` -> expands to checked prop + onChange$ handler
       - `q:slot` -> preserved as-is in the props
       - `ref` -> handled specially (becomes a _jsxSorted/Split argument)
       - `children` -> extracted from props, passed as last positional argument
       - `key` -> extracted from props, passed as key argument
    5. **Key counter:** When no explicit key is provided, generate a deterministic counter-based key
    6. **Flags:** Bitmask indicating: has dynamic children, is SVG, is void element, etc.
    7. **Event handlers in JSX props:** `onClick={$(() => ...)}` -- the `$()` inside is detected by dollar detection and goes through normal QRL extraction. The JSX transform just passes through the QRL wrapper result.

    Implement as a module with functions called from QwikTransform's `exit_jsx_element` / `exit_jsx_fragment` hooks. Add these hooks to the Traverse implementation in transform.rs.

    **inlined_fn.rs -- CONV-07: Signal Optimization**

    For inline expressions in JSX props that reference signals, generate `_fnSignal()` wrappers instead of extracting full QRL segments:

    1. **Decision flow:**
       - Is the expression a simple member access? (e.g., `store.count`) -> `_wrapProp(store, "count")`
       - Is the expression a computed expression? (e.g., `store.count + 1`) -> `_fnSignal((p0) => p0.count + 1, [store], "p0.count+1")`
       - Is the expression a constant? -> Leave as-is (no signal optimization)
       - Does the expression have side effects? -> Leave as-is (cannot optimize)
    2. **Parameter extraction:** For `_fnSignal`, extract captured variables as positional parameters (`p0`, `p1`, etc.)
    3. **Display string:** Third argument is a human-readable representation of the expression
    4. **Integration with is_const.rs:** Use `is_const_expression()` to determine if an expression is constant

    Implement as a module with functions called from the JSX transform when processing prop values.

    **Integration with transform.rs:**
    - Add `exit_jsx_element` and `exit_jsx_fragment` hooks to QwikTransform's Traverse impl
    - These hooks call into jsx_transform.rs functions
    - JSX transform calls inlined_fn.rs for signal optimization on prop values
    - Track import needs: `needs_jsx_sorted_import`, `needs_jsx_split_import`, `needs_fn_signal_import`, `needs_wrap_prop_import`

    Update `lib.rs` to add both modules.
