**#### _jsxSorted vs _jsxSplit Branch Point (per D-18)**

Document as ONE mechanism with a branching condition:
- Both produce the same 6-argument signature: `(type, varProps, constProps, children, flags, key)`
- _jsxSorted: No spread props AND no component bind:* props. Props are sorted alphabetically by key at compile time. This is the "compile-time sorted" path.
- _jsxSplit: Spread props exist OR component has bind:* props. Props are NOT sorted at compile time; runtime handles ordering. This is the "runtime-sorted" path.
- Note: SWC internal variable `should_sort` is true for _jsxSplit (counterintuitively). The spec uses clear terminology instead.
- Dev mode: Both paths add a 7th argument with source location info in Dev/Hmr emit modes.

Rules:
1. Input is jsx(type, propsObject, key?) from Stage 1 transpilation
2. Output is _jsxSorted(type, varProps, constProps, children, flags, key) when no spreads and no component bind:* props
3. Output is _jsxSplit(type, varProps, constProps, children, flags, key) when spreads exist or component has bind:* props
4. varProps sorted alphabetically by key for _jsxSorted path; unsorted for _jsxSplit
5. Empty varProps = null (2nd arg), empty constProps = null (3rd arg)
6. Dev/Hmr mode adds 7th argument with source location

**#### Element Transformation**

Document element type detection (transform.rs:1158-1174):
- String literal (e.g., "div"): is_fn=false, check is_text_only(), name_token=true
- Identifier (e.g., MyComponent): is_fn=true, is_text_only=false, name_token=true, mutable unless in immutable_function_cmp
- Other expression: is_fn=true, is_text_only=false, name_token=false, always mutable
- Text-only element list: text, textarea, title, option, script, style, noscript
- is_fn determines: key generation behavior, className handling, event name conversion behavior

**#### Prop Classification (Static vs Dynamic)**

Document the static/dynamic split (transform.rs:2066-2653):
- const_props (static): literals, template literals with only literal parts, imported identifiers, const-declared variables with static initializers
- var_props (dynamic): everything else — signal accesses, computed expressions, function calls
- Static determination via is_const_expr(): literal values, identifiers in const_idents list (IdentType::Var(true)), global identifiers, template literals where all expressions are const
- Empty bucket handling: empty var_props -> null, empty const_props -> null
- Spread interaction preview: When spreads exist, all props before the last spread go to var_props regardless of static nature (detailed in Spread Props subsection)

**#### Special Attributes**

Document ALL special attributes in a catalog table (per research lines 142-157), then detail each:

1. **children**: Extracted to 4th argument, never in props. Single child passed directly, array as JS array literal, no children -> null.
2. **ref**: Always goes to var_props (never optimized for signals)
3. **q:slot**: Always goes to var_props (never optimized for signals)
4. **className**: Renamed to "class" for native elements (is_fn=false). Kept as "className" for components (is_fn=true).
5. **class**: Kept as-is for both native and components.
6. **bind:value**: On native elements without spreads: expands to value prop + q-e:input handler using inlinedQrl(_val, "_val", [signal]). On native elements with spreads (_jsxSplit path): bind in var_props left untouched, only const_props-targeted bind expands. On components: forces _jsxSplit, no expansion.
7. **bind:checked**: Same pattern as bind:value but uses _chk helper instead of _val, and generates "checked" prop.
8. **bind:* (other)**: Unrecognized bind directives kept as-is in const_props. On components, any bind:* forces _jsxSplit.
9. **on*$ event handlers**: Arrow/function expressions extracted as QRL segments. On native elements: event name converted (onClick$ -> q-e:click). On components: $-suffix replaced with Qrl (onClick$ -> onClickQrl).
10. **q:p / q:ps**: Injected for iteration variable lifting. q:p for single variable (value is variable directly), q:ps for multiple (value is array).

**Event Name Conversion** (sub-subsection):
- on*$: prefix q-e: (onClick$ -> q-e:click)
- document:on*$: prefix q-d: (document:onScroll$ -> q-d:scroll)
- window:on*$: prefix q-w: (window:onResize$ -> q-w:resize)
- host:* prefix: NOT converted, kept as-is
- Case rules: remove $, strip scope prefix, if starts with - use as-is (case-sensitive marker), otherwise lowercase, convert camelCase to kebab-case
- Handler merging: multiple handlers for same converted key merged into array

**bind:value/bind:checked Expansion** (sub-subsection):
- bind:value={signal} expands to: value prop with signal ref in const_props + q-e:input handler: inlinedQrl(_val, "_val", [signal]) in const_props
- bind:checked={checked} expands to: checked prop + q-e:input handler: inlinedQrl(_chk, "_chk", [checked])
- Merging with existing handlers: if q-e:input already exists, bind handler merged into array

**#### Children Handling**

Rules:
1. Children extracted from props object, placed as 4th argument to _jsxSorted/_jsxSplit
2. Single child: passed directly as 4th argument
3. Array of children: passed as JS array literal
4. No children: 4th argument is null
5. Text-only elements (text, textarea, title, option, script, style, noscript): children forced to raw content treatment
6. Children in spread context: children appearing before a spread go to var_props as regular children key-value prop instead of being extracted to 4th argument
7. Child expressions referencing signals processed through convert_to_signal_item (may wrap in _fnSignal or _wrapProp — see Signal Optimization CONV-07)

**#### Key Generation**

Rules:
1. Explicit keys: If original jsx() call has 3rd argument (explicit key), used as-is
2. Auto-generated key format: "{base64_prefix}_{counter}" where base64_prefix = first 2 chars of base64(file_hash), counter = jsx_key_counter incremented per key in file
3. Example key values: "u6_0", "u6_1", "u6_2"
4. Components (is_fn=true): Always get a key (auto-generated if not explicit)
5. Root-level JSX (root_jsx_mode=true): Gets a key
6. Nested native elements in non-root position: key is null
7. root_jsx_mode starts true, reset to false after first JSX element processed, restored after processing

Example: Key Generation Across Elements (example_jsx_keyed) — Read `/Users/scottweaver/Projects/qwik-oxc-optimizer/swc-snapshots/example_jsx_keyed.snap`.

"See also": example_jsx_keyed_dev.snap (key + dev mode 7th argument)

**#### Flag Computation**

Document the flags bitmask (5th argument):

| Bit | Value | Name | Meaning |
|-----|-------|------|---------|
| 0 | 1 | static_listeners | All event handlers are const (no captured mutable state) |
| 1 | 2 | static_subtree | Children subtree is fully static |
| 2 | 4 | moved_captures | Element has q:p or q:ps props (lifted iteration variables) |

Common flag values:
- 3 = static_listeners + static_subtree (fully static element)
- 1 = static_listeners only (dynamic children)
- 0 = fully dynamic

Rule: When spreads exist, static_listeners and static_subtree both start as false (flags default to 0 in spread context).

**#### Spread Props Handling**

Rules:
1. Identifier-based spreads ({...props}): Split using _getVarProps(props) and _getConstProps(props) helpers imported from @qwik.dev/core. _getVarProps spread goes to var_props, _getConstProps spread goes to const_props.
2. Non-identifier spreads ({...getProps()}): entire spread goes to var_props as-is
3. Multiple spreads: all const_props spreads after the first spread go to var_props to maintain correct override order
4. Props before last spread: ALL regular props before the last spread go to var_props regardless of static nature
5. Props after last spread: follow normal static/dynamic classification unless has_var_prop_after_last_spread
6. Spread presence forces _jsxSplit path

Example: Spread with Additional Props (should_split_spread_props_with_additional_prop) — Read the snapshot.

**Iteration Variable Lifting (q:p/q:ps)** (sub-subsection under Spread or Special Attributes — place where it flows best):
1. In loop context: iteration variables from iteration_var_stack used by event handlers are collected
2. Outside loop context: union of all captures from all event handlers computed
3. Variables injected as extra parameters to event handler functions
4. q:p (single variable) or q:ps (multiple) prop added to var_props
5. Props referencing iteration variables moved to var_props even if otherwise const
6. sets moved_captures flag (bit 2, value 4)
