Add a "## Variable Migration" section under "## Stage 5: Post-Transform" per D-12. This is a SEPARATE section that cross-references Capture Analysis for dependency data, NOT a subsection of Capture Analysis.

**Behavioral Rules — 5-step pipeline:**

1. **`analyze_root_dependencies(module, global_collect)`** (dependency_analysis.rs):
   - Walks every top-level statement in the module
   - For each variable declaration: records which other identifiers appear in its initializer expression
   - Builds a `root_deps: HashMap<Id, RootDep>` where `RootDep` contains:
     - `deps`: set of other root-level identifiers this var depends on
     - `is_user_exported`: whether the var is exported by user code (not synthetic `_auto_` exports)
   - Skips import declarations and synthetic exports

2. **`build_root_var_usage_map(segments, root_deps)`**:
   - For each segment, check which root vars appear in the segment's `local_idents` or `scoped_idents`
   - Result: `usage_map: HashMap<Id, Vec<SegmentId>>` — which segments reference each root var
   - A root var used by 0 segments is "orphaned" (only main module uses it)
   - A root var used by exactly 1 segment is a migration candidate

3. **`build_main_module_usage_set(module, root_deps)`**:
   - Find root vars still referenced by non-declaration, non-import statements in the main module
   - These vars cannot be migrated even if only one segment uses them — the main module still needs them
   - Example: `const x = 5; console.log(x); $(() => use(x))` — `x` is used by main AND a segment, so not migratable

4. **`find_migratable_vars(segments, root_deps, usage_map, main_usage)`**:
   A variable is migratable if ALL of these conditions hold:
   - Used by exactly one segment (appears in only one segment's locals/captures)
   - NOT exported by user code (synthetic `_auto_` exports are OK to migrate)
   - NOT an import (imports stay in root and are re-emitted per segment)
   - NOT referenced by main module statements (beyond its own declaration)
   - Transitive dependencies are ALSO migratable (if var A depends on var B, both must be migratable to the same segment)
   - **Safety filter**: No root declaration OUTSIDE the target segment depends on this var (prevents breaking other root-level code)

5. **Post-migration cleanup** (parse.rs:400-435):
   - `remove_migrated_exports()`: Removes synthetic `_auto_` exports for migrated vars from root module
   - Second DCE pass: Runs if any vars were migrated, to remove now-unused imports that only the migrated var needed
   - The migrated var declarations are added to the segment module via `migrated_root_vars` field (consumed by `new_module` in code_move.rs)

**Source reference:** dependency_analysis.rs (full file), parse.rs:400-435 (migration invocation + cleanup). Cross-references: Capture Analysis (scoped_idents), Segment Extraction (local_idents, migrated_root_vars).

**Examples per D-14 (2 examples — variable migration is an optimization, edge cases are more important than basic behavior):**
1. "Basic Variable Migration": Root module has `const helper = (x) => x * 2;` used only by one segment. Show before (helper in root, self-imported by segment) and after (helper moved INTO segment, no self-import needed, root module smaller).

Also show transitive case: if `const derived = CONFIG.x` is used by only one segment but depends on CONFIG which is shared, `derived` is NOT migratable.

Add a "### Source Map Generation" section under the Infrastructure area of the spec (alongside Hash Generation, Path Resolution, GlobalCollect).

**Behavioral Rules:**

1. **Generation mechanism** (`emit_source_code` in parse.rs:701-753):
   - Creates a `JsWriter` that records byte-position-to-original-position mappings during JavaScript serialization
   - Runs `Emitter::emit_module()` which produces JavaScript source code string + a source mapping buffer
   - Builds a V3-format source map from the recorded mappings
   - Optionally sets `sourceRoot` from the `root_dir` config option
   - Returns `(code: String, map: Option<String>)` where `map` is the JSON-stringified source map (None if `source_maps: false`)

2. **Root module source maps**:
   - Computed from the mutated AST (after all transforms)
   - Span positions in the AST reference original source locations tracked by SWC's SourceMap
   - Mapping quality is HIGH — the root module AST is mutated in-place, so spans are preserved from the original parse
   - The source map's `sources` array contains the original file path
   - The `sourcesContent` array contains the original source code (if available)

3. **Segment module source maps**:
   - Each segment module is independently constructed via `new_module()`, then run through hygiene + fixer + codegen with source map generation
   - The segment AST is built from scratch (not cloned from original) — it is a new `Module` node assembled from extracted function bodies and generated import statements
   - Source map fidelity varies: function body spans are preserved from the original parse (so they map back correctly), but generated imports/exports and `_captures` destructuring have no original-source mapping
   - The source map's `sources` array contains the ORIGINAL file path (not the segment filename)

4. **Configuration**:
   - `source_maps: bool` in TransformModulesOptions — when false, `map` field is None for all modules
   - Each `TransformModule` in the output has a `map: Option<String>` field containing the source map JSON

5. **Behavioral contract**: Source maps SHOULD reference original file positions. For root modules, this is reliable. For segment modules, function body mappings are reliable but synthetic code (imports, captures destructuring, export wrapper) has no original-source mapping.

**Source reference:** parse.rs:701-753 (emit_source_code)
