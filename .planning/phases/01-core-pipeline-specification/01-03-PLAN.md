**Algorithm Overview (4 steps):**
1. **Collect descendant identifiers**: `IdentCollector` visits the `$()` callback body via SWC's Visit trait. Collects all referenced identifiers. Filters: SyntaxContext must not be empty, excludes global names (`undefined`, `NaN`, `Infinity`, `null`). Uses `ExprOrSkip` enum to skip JSX attribute names while still collecting JSX expression containers. Source: collector.rs:400-530.
2. **Partition declaration stack**: `decl_stack` entries accumulated during the AST traversal are partitioned into `Var`-type declarations (capturable — let/const/var/param/catch) and non-`Var` declarations (function/class — these produce ERROR diagnostics if referenced across `$()` boundaries). Source: transform.rs:960-975.
3. **Compute scoped identifiers**: `compute_scoped_idents()` (transform.rs:4894) performs set intersection: identifiers that appear in BOTH the descendant set (from step 1) AND the `Var`-type declaration stack entries. These are the captured variables. Function callback parameters are excluded (they are local to the segment).
4. **Classify each identifier** against GlobalCollect: if `global_collect.is_global(id)` returns true, the identifier is NOT a capture — it will be resolved via imports in the segment module. Only identifiers that are local to an enclosing scope (not global, not parameters of the `$()` callback) become captures.

**Mermaid Decision Tree (per D-09):**
Create a Mermaid flowchart showing the classification logic for a variable reference inside a `$()` body:
```
Start: "Variable V referenced in $() body"
  -> Is V a parameter of the $() callback? -> YES: "Not captured (local to segment)"
  -> Is V in global_collect.imports? -> YES: "Category 2: Re-emitted import"
  -> Is V in global_collect.exports or global_collect.root? -> YES: "Category 1: Self-import"
  -> Is V a TypeScript type-only import? -> YES: "Category 6: Erased"
  -> Is V's declaration a function/class? -> YES: "Category 8: ERROR"
  -> Is V shadowed by an inner binding in $() body? -> YES: "Category 7: Inner wins"
  -> Is V in an iteration scope (for-of/for-in/for)? -> Goes to Category 4
  -> Is V in decl_stack as Var-type? -> YES: "Category 3: Captured (outer local)"
  -> Otherwise: "Not referenced / global"
```

**Self-Import Reclassification subsection:**
Document explicitly (this fixed 46 of Jack's 293 runtime deviations):
1. When a segment references a module-level declaration (const/function/class/enum at root scope), it does NOT become a `_captures[N]` entry
2. Instead, `ensure_export()` (transform.rs:1024-1026) adds a synthetic `_auto_{name}` named export to the root module
3. `code_move::new_module()` resolves this identifier via `resolve_export_for_id()` and generates `import { _auto_X as X } from "./module_stem"` in the segment
4. The segment's `captures` field in SegmentAnalysis is `false` for these identifiers
5. The `_auto_` prefix prevents collision with user exports
6. Source: transform.rs:1024-1026 (ensure_export), code_move.rs:200-276 (resolve_export_for_id)

**Important behavioral note**: "Capture analysis proceeds regardless of diagnostic errors. Only bail if the parsed body is empty. Semantic errors (e.g., `await` in non-async) produce valid ASTs that can still be analyzed." (from Pitfall 6 in research)
