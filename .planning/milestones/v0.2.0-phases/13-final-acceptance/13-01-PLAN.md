  <action>
The SWC output preserves original quote style for user-written import declarations. OXC Codegen normalizes all string literals to double quotes. The fix:

Add a `preserve_original_quotes` function to emit.rs that:

1. Takes the codegen output string AND the original source string as parameters
2. For each line in the output that starts with `import ` and contains a `from ` clause:
   - Extract the module specifier (the string after `from`)
   - Search the original source for the same import specifier
   - If the original source uses single quotes for that specifier, replace double quotes with single quotes in the output line
3. Only applies to import declarations (lines starting with `import `), NOT to other string literals in the code

Implementation approach:
- Parse each import line to find the `from "..."` or `from '...'` portion
- For each import specifier found in the output, search the original source for the same specifier
- If the original uses `'specifier'`, convert the output line's `"specifier"` to `'specifier'`
- This is a post-processing step, NOT an AST-level change

Call this function in `emit_module` after `normalize_pure_annotations` and before `insert_separator_comments`:
```rust
let code = normalize_pure_annotations(&codegen_result.code);
let code = preserve_original_quotes(&code, source);
let code = insert_separator_comments(&code);
```

Update the `emit_module` signature: the `source` parameter is already available (it's the 2nd argument).

IMPORTANT: Only change quotes on import lines where the ORIGINAL source used single quotes. Synthesized imports (generated by the optimizer, not present in original source) should keep double quotes (matching SWC behavior).

Add a unit test `test_preserve_original_quotes` that verifies:
- Input with `import { x } from "@qwik.dev/core"` where source has `from '@qwik.dev/core'` outputs single quotes
- Input with `import { qrl } from "@qwik.dev/core"` where source does NOT contain that import keeps double quotes
  </action>
